Issues Found (from phase5.md review)
-------------------------------------
Status key: [FIXED] [OPEN] [WONTFIX]

Issue 1 - [WONTFIX] Step 5.3 strips space before punctuation inside markers
  Phase: 5
  Severity: Low
  Detail: Step 5.3 regex `re.sub(r" ([.,;:!?])", ...)` removes space before
    punctuation globally, including inside marker boundaries. On review, this
    is actually correct behavior -- the spaces before punctuation are PDF
    extraction artifacts and the cleaned output reads better.

Issue 2 - [FIXED] Headings and body text on same line
  Phase: 4
  Severity: High
  Root cause: phase4.py:259 used `" ".join(block_text_parts)` to join lines
    within a PDF block. A PDF block can contain a heading line followed by
    body lines, so joining with space collapsed them onto one line.
  Fix: Changed to `"\n".join(block_text_parts)`. Each block_text_parts entry
    represents a separate PDF line from PyMuPDF, and these are genuine layout
    boundaries.
  Files: src/gm_kit/pdf_convert/phases/phase4.py (line 259)

Issue 3 - [FIXED] Empty icon-font markers
  Phase: 4 or 5
  Severity: Medium
  Detail: FontAwesome glyphs (sig010, sig017, sig018) render as empty marker
    content: `<<sig010:>>`. These are icon characters (U+F0FE, U+F15B, U+F043)
    that PyMuPDF extracts as private-use-area codepoints or empty strings.
    They clutter the output and carry no textual meaning.
  Examples:
    Line 20: <<sig010:>><<sig009: in the navbar>>
    Line 42: <<sig017:>><<sig015: A4 Pagesize>>
  Fix: Strip markers with empty or whitespace-only content, or
    strip markers whose signature maps to a known icon font family.

Issue 4 - [FIXED] Whitespace-only marker lines
  Phase: 4/5
  Severity: Low
  Detail: Lines containing only `<<sig009: >>` (a single space in a marker)
    appear as noise lines with no content value.
  Examples: Lines 96, 100, 101, 102, 205, 206
  Fix: Strip lines that contain only whitespace-bearing markers.

Issue 5 - [FIXED] Page numbers left as content
  Phase: 4
  Severity: Medium
  Detail: PDF page numbers are extracted as regular text content.
  Examples:
    Line 99:  <<sig019:1>>
    Line 207: <<sig019:2>>
  Fix: Detect page-number signatures by font characteristics (small size, digit-only sample, not a heading) and strip them.

Issue 6 - [FIXED] Footer/watermark text left inline
  Phase: 4
  Severity: Medium
  Detail: Decorative elements like watermarks and page footers are preserved
    as document content.
  Examples:
    Line 98:  <<sig021:naturalcrit>>   (watermark/branding)
    Line 103: <<sig020:PART 1 | FANCINESS>>  (page footer)
    Line 208: <<sig020:PART 2 | BORING STUFF>>  (page footer)
  Note: The PART footers could be confused for headings by Phase 8.
  Fix: Identify footer/watermark signatures by font characteristics (very small size, specific keywords in samples) and strip them.

Issue 7 - [DEFERRED] Table data collapsed
  Phase: 4
  Severity: Low (known limitation)
  Detail: Table cell data is extracted as individual lines with no structural
    information. Table headers and data cells appear as a flat list.
  Example (lines 166-181): Head A, Spanned Header, Head B, ... 1A 2A 3A, 1B, ...
  Note: Reconstructing table structure from PDF text extraction alone is very
    difficult without spatial analysis. This may need to be an agent step or
    a future enhancement.
  Deferral Rationale: This issue requires complex structural/spatial analysis outside the scope of Phase 5. A separate backlog item for table placeholder insertion will be created for Phase 7: Structural Detection, and a discussion of multimodal OCR for tables will be added to pdf-conversion-architecture.md as a future option.
